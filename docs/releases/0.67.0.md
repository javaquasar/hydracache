# HydraCache 0.67.0 - Performance Characterization and Capacity Baselines

Status: **in progress**. The W0-W10 implementation is closed on the release
branch, but this file is candidate documentation, not a shipped-release or
performance claim.

## Scope implemented

Release 0.67 adds a development-only, coordinated-omission-safe measurement
toolchain over the surfaces that actually exist:

- W0 provides fixed-rate open-loop scheduling, scheduled-send latency,
  bounded histograms, the complete sustainable-knee predicate, repeat/reset
  discipline, runner qualification, and typed reports.
- W1 characterizes the real embedded local cache and reuses the existing trace
  catalog rather than creating a second workload authority.
- W2 characterizes the real in-process Axum client router. It is not a daemon,
  socket, or native-wire benchmark.
- W3 measures one real loopback RESP endpoint on one prebuilt daemon. Its value
  state is node-local; `redis-benchmark` is supplemental closed-loop evidence.
- W4 keeps real 3/5/7-daemon control-plane/admin measurements separate from
  in-process grid/model primitive costs.
- W5 records separate control-plane, killed RESP endpoint, and model brownout
  profiles; it does not invent one blended cluster failover number.
- W6 derives overload goodput/recovery only from a valid surface-scoped knee.
- W7 implements immutable-anchor plus rolling-`main` budget governance and
  forbids silent rebaseline.
- W8 runs one pinned, alternating-order, same-box Redis comparison with the
  same tool and workload. It is evidence for that method only, not marketing.
- W9 cross-checks only metrics already exported by the real daemon. Missing
  fields remain `not_available`; no metric is added to make the check pass.
- W10 binds profiles, prebuild identities, scenarios, reports, canaries,
  registered commands, CI lanes, and exact-candidate receipts.

The load generator remains `publish = false`, and the release adds no product
surface, protocol, consistency level, or optimization.

## Claim boundaries

The release does not measure or claim:

- a mounted `/client/v1/*` daemon listener;
- distributed or cross-daemon RESP values, locks, TTLs, or failover;
- summed RESP throughput as cluster capacity;
- end-to-end distributed value-grid or live-reshard performance;
- universal HydraCache-versus-Redis superiority; or
- in-process instrumentation relabeled as daemon exporter evidence.

See [`../PERFORMANCE.md`](../PERFORMANCE.md) for the complete surface matrix and
quoting rules.

## Implementation closure is not ship closure

The annotated predecessor `v0.66.0` is present, its roadmap entry is `shipped`,
and the tag is an ancestor of this branch. The candidate remains intentionally
**no-ship** as of 2026-07-21 because:

1. The committed W7 `reference-v1` anchor, rolling baseline, and numerical
   budgets are `unbootstrapped`.
2. Bootstrap requires at least five eligible successful serialized GitHub-hosted
   `main` runs from the same pinned image/contract family, followed by independent
   review of the exact anchor, window, and budget payload. A candidate cannot
   baseline or approve itself.
3. After the bootstrap blocker is resolved, all mandatory
   core, RESP/Redis, control-plane, budget, and canary gates must be rerun on one
   frozen clean candidate. Any later code, scenario, budget, registry, or docs
   change invalidates those receipts.

Automatic `ci-shared` runs remain broad regression tripwires only. Explicit serialized
`reference-v1` runs on pinned `ubuntu-24.04` may bootstrap and satisfy the relative
same-image release gate; they do not establish portable capacity floors or sizing.

This hosted profile is explicitly four-core: W1 records only the `1/2/4` scaling
series and W4B uses the same four-core attestation. No eight-core point or extrapolation
is part of the 0.67 hosted evidence.

Ship status is determined only by:

```text
cargo run -p xtask --locked -- release-evidence --release 0.67 --receipts-dir target/release-evidence/receipts --require-ship
```

Until that exact-candidate command is green, HydraCache publishes no official
`0.67.0` capacity floor, sizing guidance, Redis comparison, or metrics-agreement
result.
