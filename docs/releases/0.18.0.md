# HydraCache 0.18.0

Status: in development.

## Cache Events And Listeners

`HydraCache` now exposes a first-class event subscription API:

- `HydraCache::subscribe(CacheEventOptions)`
- `HydraCache::subscribe_mutations()`
- `HydraCache::subscribe_access()`
- `HydraCache::subscribe_key(...)`
- `HydraCache::subscribe_tag(...)`
- `HydraCache::add_listener(...)`
- `HydraCache::on_mutation(...)`
- `HydraCache::on_access(...)`
- `TypedCache::subscribe(CacheEventOptions)`
- `TypedCache::subscribe_namespace()`
- `TypedCache::subscribe_key(...)`
- `TypedCache::subscribe_mutations()`
- `TypedCache::subscribe_access()`
- `TypedCache::subscribe_tag(...)`
- `CacheEvent`, `CacheEventKind`, `CacheEventScope`, `CacheEventOrigin`
- `CacheEventSubscriber::recv()`
- `CacheEventSubscriber::next_event()`
- `CacheEventListenerHandle`

Mutation and invalidation events are available by default when subscribers
exist. High-volume hit/miss/load events are opt-in with
`HydraCache::local().enable_access_events(true)`.

## Runtime Guarantees

- Event publication never waits for subscribers.
- Subscribers use a bounded ring buffer configured with
  `event_buffer_capacity`.
- Slow subscribers receive `CacheEventRecvError::Lagged`.
- Events are metadata-only. Cached values are not exposed through events.

## Tests

The new event module covers mutation events, access events, filters, typed cache
delegation, single-flight joins, stale-load discards, loader failures, and
subscriber lag.

`hydracache` also includes performance smoke tests for hot-hit behavior,
single-flight loader deduplication, warmed multi-key workloads, and bulk tag
invalidation. The tests assert stable performance properties without relying on
machine-specific latency thresholds.

An ignored allocation profile harness was added for manual before/after checks
around hot hits, metadata-only hits, typed-cache hits, and tag invalidation. The
allocation optimization plan lives in
`docs/plans/V0_18_ALLOCATION_OPTIMIZATION_PLAN.md`.

## Sandbox

`hydracache-sandbox` now exposes `POST /demo/listeners/run` for manual and
Swagger-based listener checks. The scenario subscribes to mutation, access, key,
tag, and callback listeners, runs a put -> hit -> tag invalidation -> reload
flow, and returns the captured listener events plus the matching sandbox event
log entry.
