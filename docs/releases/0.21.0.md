# HydraCache 0.21.0

Status: in development.

0.21.0 focuses on cluster operability: stronger post-publish verification,
richer diagnostics, the first ownership resolver API, a transport-neutral peer
fetch seam, and a sandbox lab that makes the cluster flow easier to explain.

## Planned Scope

- Post-publish verification hardening for every published crate.
- Cluster diagnostics v2.
- Deterministic ownership resolver.
- Peer fetch design/API spike.
- Sandbox cluster lab scenario.
- Release hardening and documentation updates.

## Post-Publish Verification Hardening

The post-publish workflow now verifies the published crates as an external
consumer would use them. The smoke crate pulls `hydracache`,
`hydracache-core`, `hydracache-cluster`, `hydracache-cluster-chitchat`,
`hydracache-cluster-raft`, `hydracache-observability`,
`hydracache-actuator-axum`, `hydracache-db`, and `hydracache-sqlx` from
crates.io and exercises:

- local typed cache APIs;
- cluster composition through `HydraCluster`;
- real chitchat discovery and raft metadata crates;
- DB cache adapter aliases;
- framework-neutral observability snapshots;
- Axum actuator route construction.

## Cluster Diagnostics v2

`ClusterDiagnostics` now includes cheap derived helper methods for dashboards,
actuator-style reports, and sandbox scenarios:

- `is_local_role()`, `is_client_role()`, and `is_member_role()`;
- `participant_count()` and `bootstrap_count()`;
- `has_members()`, `has_clients()`, and `has_bootstrap()`;
- `has_invalidation_subscribers()` and `has_membership_subscribers()`;
- `has_multiple_participants()`;
- `is_operational()`.

The sandbox cluster runtime reports include the same derived values in JSON, so
manual demos can distinguish "connected but alone" from a real multi-runtime
cluster view.

## Ownership Resolver

0.21.0 adds the first deterministic ownership primitive:

- `ClusterOwnershipResolver`
- `RendezvousClusterOwnership`
- `ClusterOwnershipDecision`
- `InMemoryCluster::owner_for_key(...)`
- `InMemoryCluster::owner_for_key_with(...)`

The default resolver uses stable rendezvous-style hashing over the logical key
and admitted member node ids. It ignores clients, returns no owner when no
members are admitted, and performs no network calls or value loading. This is
the foundation for a later peer-fetch layer without changing the local cache
hot path.

## Peer Fetch API Spike

0.21.0 adds a small, transport-neutral peer-fetch seam:

- `ClusterPeerFetchRequest`
- `ClusterPeerFetchResponse`
- `ClusterPeerFetch`
- `InMemoryPeerFetch`
- `ClusterOwnershipDecision::peer_fetch_request()`

The API moves encoded bytes, not typed values. This keeps it independent of
SQLx, Diesel, SeaORM, custom codecs, and future network transports. The
in-memory implementation is useful for tests, demos, and sandbox scenarios; it
does not perform network IO or owner-side loader execution.

## Sandbox Cluster Lab

The sandbox now includes `POST /demo/cluster/ownership/run`. The scenario
creates two member runtimes and one client runtime on an in-memory cluster,
resolves a deterministic owner for a key, performs an in-memory peer fetch of
encoded bytes from the selected owner, and then verifies that owner-originated
tag invalidation reaches the client near-cache.

The response includes:

- the ownership decision and resolver metadata;
- peer-fetch hit/miss details and decoded UTF-8 preview when possible;
- member/client diagnostics with derived operational flags;
- a short timeline covering admission, ownership, peer fetch, and invalidation;
- a `passed` flag suitable for manual release checks and dashboard demos.

## Non-Goals

- Production multi-node Raft transport.
- Durable metadata storage.
- Remote owner-side query execution.
- External invalidation transports.
- Production-grade security/authentication.

See [the 0.21.0 plan](../plans/V0_21_CLUSTER_OPERABILITY_PLAN.md) for the
implementation roadmap.
