# HydraCache 0.16.0 Release Notes

HydraCache `0.16.0` adds local observability helpers and optional read-only
actuator modules so users can quickly confirm that a cached path is producing
misses, loads, and hits as expected.

## Added

- `CacheStats::total_requests()`.
- `CacheStats::hit_ratio()`.
- `CacheStats::has_single_flight_activity()`.
- `CacheStats::has_stale_load_discards()`.
- `CacheDiagnostics`.
- `HydraCache::diagnostics().await`.
- `TypedCache::diagnostics().await`.
- `hydracache-observability`.
- `hydracache-actuator-axum`.
- `HydraCacheRegistry` for named cache registration.
- `HydraCacheProbe` for adapting a `HydraCache` instance into the registry.
- Serializable `CacheStatsSnapshot`.
- Serializable `CacheDiagnosticsSnapshot`.
- Serializable `HydraCacheOverview`.
- Read-only Axum routes for health, cache list, overview, diagnostics, and
  stats.
- Non-published `hydracache-sandbox` manual backend with memory, SQLite memory,
  SQLite file, Docker Compose Postgres, and Postgres Docker modes.
- Committed `crates/hydracache-sandbox/.env` demo profile plus
  `.env.example` reference for sandbox defaults.
- Sandbox profile presets through `HYDRACACHE_SANDBOX_PROFILE` and `--profile`.
- Docker Compose files for running sandbox Postgres dependencies or the full
  sandbox stack.
- Unified Compose profile file plus `Dockerfile.sandbox` for a prebuilt local
  sandbox API image.
- Generated sandbox OpenAPI JSON at `/openapi.json` through `utoipa`.
- Local embedded Swagger UI assets at `/swagger-ui` through
  `utoipa-swagger-ui`, with no CDN dependency.
- Local no-CDN developer console at `/demo/ui`.
- Expanded sandbox Swagger API that can reproduce raw local-cache operations,
  typed-cache namespacing, database-backed query caching, cached non-database
  functions, TTL expiry, single-flight, and invalidation/load race safety.
- Application-level sandbox report at `/demo/report` with active profile,
  backend, loader counters, function counters, event count, capabilities, and
  cache diagnostics.
- Structured event log at `/demo/events`, reset endpoint at `/demo/reset`,
  readiness endpoint at `/ready`, and expected-failure negative scenarios under
  `/demo/negative/*`.
- Runtime sandbox config at `/demo/config` and copyable scenario presets at
  `/demo/presets`.
- Event log filtering by `kind`, `key`, `tag`, `flow_id`, and `limit`.
- Optional append-only JSONL event persistence through
  `HYDRACACHE_SANDBOX_EVENT_LOG_PATH` or `--event-log-path`.
- Portable sandbox export bundle at `/demo/export`.
- Built-in sandbox self-test at `POST /demo/self-test` with step-level results
  and a correlated event log.
- Scenario runner at `POST /demo/scenarios/run` for `golden-path`, `ttl`,
  `single-flight`, `invalidation-race`, `negative-suite`, and `self-test`.
- Flow timeline endpoint at `GET /demo/flows/{flow_id}/timeline`.
- Profile comparison endpoint at `POST /demo/profiles/compare` for supported
  local profiles, with Postgres profiles reported as skipped instead of
  implicitly requiring external services.
- Replay endpoint at `POST /demo/replay`, fault-injection endpoint at
  `POST /demo/faults/run`, and manual benchmark endpoint at
  `POST /demo/benchmarks/manual`.
- Optional local bearer-token guard through `HYDRACACHE_SANDBOX_TOKEN` or
  `--token`, with guard status at `GET /demo/security`.
- Latency summaries on event logs, reports, exports, timelines, scenario
  responses, and manual benchmark responses.
- No-CDN developer console counters for hits, misses, loads, and single-flight
  joins.
- Docker Compose healthchecks for the full sandbox API service.
- Sandbox HTTP collection and PowerShell demo scripts.
- Optional Postgres Docker smoke test with graceful skip when Docker is
  unavailable.
- README and cargo-doc examples showing how to verify a repeated cached call.

## Example

```rust
use hydracache::{CacheOptions, HydraCache};

# async fn example() -> hydracache::CacheResult<()> {
let cache = HydraCache::local().build();

let first = cache
    .get_or_insert_with("answer", CacheOptions::new(), || async { 42_u64 })
    .await?;
let second = cache
    .get_or_insert_with("answer", CacheOptions::new(), || async { 7_u64 })
    .await?;

let diagnostics = cache.diagnostics().await;

assert_eq!((first, second), (42, 42));
assert_eq!(diagnostics.stats.loads, 1);
assert_eq!(diagnostics.stats.hits, 1);
assert_eq!(diagnostics.total_requests(), 2);
assert_eq!(diagnostics.hit_ratio(), Some(0.5));
assert!(!diagnostics.is_empty());
# Ok(())
# }
```

## Axum Actuator Example

```rust
use axum::Router;
use hydracache::HydraCache;
use hydracache_actuator_axum::HydraCacheActuator;
use hydracache_observability::HydraCacheRegistry;

let cache = HydraCache::local().build();
let registry = HydraCacheRegistry::new().with_cache("main", cache);

let app: Router = Router::new().nest(
    "/actuator/hydracache",
    HydraCacheActuator::new(registry).routes(),
);
# let _ = app;
```

The actuator exposes read-only routes:

```text

GET /actuator/hydracache/caches
GET /actuator/hydracache/caches/main/diagnostics
GET /actuator/hydracache/caches/main/stats
```

## Compatibility

This release is additive. Existing `CacheStats` fields, `HydraCache::stats()`,
`TypedCache::stats()`, cacheable macros, and database adapter APIs remain
available.

The actuator is read-only. The workspace-only sandbox intentionally has
mutation endpoints for manual demos and reports, but there are no public
write-enabled actuator endpoints in this release.

```powershell
cargo +1.88.0 test --workspace --locked

cargo package -p hydracache

cargo package -p hydracache-actuator-axum
cargo package -p hydracache-db
cargo package -p hydracache-sqlx
cargo test -p hydracache-sandbox --locked
cargo test -p hydracache-sandbox --test postgres_smoke --locked
docker compose -f crates/hydracache-sandbox/compose/docker-compose.yml --profile postgres config
docker compose -f crates/hydracache-sandbox/compose/docker-compose.yml --profile full config
docker compose -f crates/hydracache-sandbox/compose/docker-compose.full.yml config
```

The dependent package verification commands should be run in publish order. For
example, `hydracache` requires `hydracache-core 0.16.0` and
`hydracache-macros 0.16.0` to be visible in the crates.io index before its
normal `cargo package` verification can resolve registry dependencies.
`hydracache-observability` requires the matching `hydracache` version, and
`hydracache-actuator-axum` requires the matching observability crate version.
`hydracache-sandbox` has `publish = false`; do not publish it.
