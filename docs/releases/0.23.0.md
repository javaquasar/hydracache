# HydraCache 0.23.0

> Status: published.

0.23.0 focuses on peer-fetch routing. `0.22.0` introduced the HTTP transport;
this release connects ownership decisions to advertised member endpoints so
callers do not need to manually build owner URLs.

## Highlights

- Stable peer-fetch endpoint metadata helpers on `ClusterCandidate` and
  `ClusterMember`.
- `PeerFetchRouter` in `hydracache-cluster-transport-axum` for
  `ownership decision -> advertised endpoint -> HTTP peer-fetch`.
- Explicit router outcomes and diagnostics for attempts, hits, misses,
  no-owner decisions, missing endpoints, generation mismatches, and transport
  errors.
- Sandbox demo at `POST /demo/cluster/routed-peer-fetch/run` for
  `resolve owner -> endpoint -> HTTP peer-fetch`.
- README, testing notes, post-publish smoke coverage, and rustdoc examples for
  advertised endpoints and routed fetch.

## Intended Shape

```rust
use hydracache::{ClusterCandidate, ClusterGeneration};
use hydracache_cluster_transport_axum::PeerFetchRouter;

# async fn example(decision: hydracache::ClusterOwnershipDecision) -> hydracache::CacheResult<()> {
let candidate = ClusterCandidate::member("member-a")
    .generation(ClusterGeneration::new(1))
    .peer_fetch_base_url("http://127.0.0.1:3000");

let router = PeerFetchRouter::new();
let outcome = router.fetch_owner_value(decision).await;
# let _ = candidate;
# let _ = outcome;
# Ok(())
# }
```

## Boundaries

The router only fetches values that already exist in the owner member cache. It
does not yet ask the owner to execute a loader or database query on miss. TLS,
authentication, failover repair, and backup ownership are still future work.

The sandbox route starts two temporary HTTP peer-fetch services, advertises
their base URLs in member metadata, resolves the owner for a key, stores encoded
bytes on the selected owner, routes the fetch through `PeerFetchRouter`, and
returns `routed_peer_fetch`, `router_diagnostics`, temporary endpoints, and a
three-step timeline.
