# HydraCache 0.20.0

Status: ready for release.

## Client And Member Cluster Shape

HydraCache now has the first public cluster role API:

- `HydraCache::client()`
- `HydraCache::member()`
- `HydraCacheClientBuilder`
- `HydraCacheMemberBuilder`
- `ClusterRole`
- `ClusterNodeId`
- `ClusterGeneration`
- `ClusterEpoch`
- `ClusterEndpoints`
- `ClusterCandidate`
- `ClusterMember`
- `ClusterDiagnostics`
- `ClusterControlPlane`
- `ClusterDiscovery`
- `ClusterDiscoveryDiagnostics`
- `InMemoryCluster`
- `InMemoryClusterDiscovery`

The goal of this release is to make the future distributed shape explicit
without forcing applications to run an external service. A client is an
application-side near-cache. A member is a cluster participant that can route
invalidation intent and later participate in authoritative metadata.

## In-Memory Cluster Runtime

`InMemoryCluster` is a test, demo, and embedded simulation runtime. It provides:

- client/member admission
- stale generation rejection for restarted node ids
- simulated membership epoch changes
- member/client snapshots
- membership events
- a shared `InMemoryInvalidationBus`
- diagnostics for role, node id, generation, epoch, bootstrap nodes, and bus
  subscribers

It does not perform network discovery, value replication, Raft consensus, or
ownership transfer. Cached values remain local; only invalidation intent is
propagated.

## In-Memory Discovery Runtime

`InMemoryClusterDiscovery` models the future chitchat/discovery side without
adding a network dependency yet. It records:

- announced candidates
- candidate role, generation, endpoints, and metadata
- live/suspect/dead liveness events
- a latest candidate snapshot by node id

Client and member builders can call `.shared_discovery(...)`, which records the
candidate before authoritative admission into `InMemoryCluster`. This keeps the
intended boundary visible:

```text
discovery observes candidates -> cluster runtime admits members/clients -> invalidation bus propagates freshness
```

`ClusterDiscovery` is the transport-neutral trait behind that model. Existing
code can use `.shared_discovery(Arc<InMemoryClusterDiscovery>)`, while future
adapters can use `.discovery(Arc<dyn ClusterDiscovery>)`.

Discovery announcement happens before control-plane admission. If discovery
fails, `connect()` or `start()` returns the error and the candidate is not
admitted.

## Diagnostics

`HydraCache::cluster_diagnostics()` returns `None` for local caches and returns
`ClusterDiagnostics` for client/member runtimes. This lets applications and the
sandbox distinguish:

- `Local`: no cluster runtime
- `Client`: near-cache connected to a cluster runtime
- `Member`: cluster participant connected to a cluster runtime

`HydraCache::cluster_discovery_diagnostics()` returns `None` for local caches
and for client/member runtimes without a discovery adapter. When discovery is
configured, it returns `ClusterDiscoveryDiagnostics` with the local node id,
latest candidates, and discovery event history.

## Cluster Lifecycle

`HydraCache::leave_cluster().await` is now available for client/member
runtimes. Local caches return `Ok(None)`.

For admitted clients and members, the method calls the configured
`ClusterControlPlane::leave(...)` with the local `ClusterGeneration` and
returns the resulting `ClusterMembershipEvent`. Leaving removes membership
metadata, but it does not clear local cache contents.

Leave is generation-safe. If an older process is still alive after a node id was
reused by a newer process generation, the older process cannot remove the newer
membership record.

Cluster-originated invalidation messages now also carry the publishing
generation. Before publishing, a client/member validates that it still owns its
admitted generation. Before applying a remote cluster message, receivers reject
messages whose source generation no longer matches the control-plane metadata.
This keeps old runtimes from publishing freshness changes after restart.

## Sandbox Cluster Demo

The workspace-only sandbox now exposes:

```text
POST /demo/cluster/lifecycle/run
```

The route creates a temporary member/client cluster using the public API,
records discovery candidates, verifies member-to-client tag invalidation and
client-to-member key invalidation, calls `leave_cluster()` for both runtimes,
and reports a visual timeline plus pass/fail assertions through OpenAPI.

## Cluster Control Plane Contract

`ClusterControlPlane` is the transport-neutral seam for future cluster
backends. It covers:

- logical cluster name
- invalidation bus selection after admission
- member admission
- client admission
- node leave
- role/node/generation/epoch diagnostics

`InMemoryCluster` implements this trait, so existing `.shared_cluster(...)`
usage remains the ergonomic in-memory path. Custom future adapters can be
passed through `.control_plane(Arc<dyn ClusterControlPlane>)`.

This is still not a network implementation. It gives future chitchat and
raft-rs adapters a stable boundary to implement without changing ordinary cache
usage.

## Chitchat-Style Discovery Adapter Spike

`ChitchatStyleDiscovery` is a dependency-free discovery adapter that models the
future chitchat path without running a networked gossip protocol yet. It keeps
seed addresses, records candidate announcements, annotates candidates with
`discovery.adapter = chitchat-style`, and records live/suspect/dead liveness
events through the existing `ClusterDiscovery` trait.

This gives the public API a seed-node, gossip-shaped discovery seam while
keeping `Local` mode free from distributed dependencies.

## Chitchat-Backed Discovery Crate

`hydracache-cluster-chitchat` is an optional crate with a real
`chitchat`-backed `ClusterDiscovery` implementation.

It provides:

- `ChitchatDiscoveryConfig`
- `ChitchatDiscovery::spawn_udp(...)`
- `ChitchatDiscovery::spawn_with_transport(...)`
- `ChitchatDiscovery::mark_leaving(...)` for generation-safe graceful-leave
  markers
- real `chitchat::spawn_chitchat(...)` runtime integration
- HydraCache candidate metadata stored in chitchat `NodeState`
- liveness/candidate snapshots exposed through the existing
  `ClusterDiscovery` trait
- deterministic tests with chitchat `ChannelTransport`

The base `hydracache` crate still does not depend on chitchat. Applications opt
in only when they need gossip-backed candidate discovery.

Graceful leave is visible in chitchat as advisory metadata:

```text
hydracache.lifecycle = leaving
hydracache.left.generation = <generation>
hydracache.left.role = member|client
```

Remote discovery snapshots expose those values as candidate metadata
`lifecycle`, `left.generation`, and `left.role`. Rejoining with a newer
generation switches the lifecycle back to `active`, while older leave markers
are rejected before they can overwrite newer node-state.

## Raft-Style Metadata Control Plane Spike

`RaftStyleMetadataControlPlane` is a dependency-free control-plane adapter that
models the future raft-rs metadata path without starting a Raft runtime yet.
Successful member/client admissions and node leaves are appended as
`RaftMetadataCommand` values, and `snapshot()` exposes term, commit index,
cluster epoch, member/client counts, and the last committed command.

Stale-generation rejections do not advance the commit index. This keeps the
spike aligned with a future replicated state machine: only accepted metadata
changes become committed commands.

## raft-rs Metadata Runtime Crate

`hydracache-cluster-raft` is an optional crate with a real raft-rs metadata
runtime behind `ClusterControlPlane`.

The current runtime is intentionally single-node and in-memory, but it uses the
actual raft-rs lifecycle:

- `raft::RawNode<MemStorage>`
- single-node campaign to leader
- metadata command proposal
- `Ready` draining
- stable log append
- committed entry application
- generation-safe leave validation
- stale-generation rejection without committing extra metadata commands
- idempotent command ids for retry-safe admission
- command result reporting through `RaftCommandResult`
- in-memory metadata snapshot export/import
- materialized membership updates only after successful Raft commit

This is not yet a multi-node networked Raft cluster. It is the real adapter
boundary and state-machine path needed before adding transport between Raft
members.

## Chitchat To Raft Admission Bridge

`ClusterAdmissionBridge` now connects discovery candidates to an authoritative
control plane. It can be used deterministically with `run_once()` in tests or
started as a background polling loop with `start()`.

The release includes integration coverage for the real optional adapter path:

```text
ChitchatDiscovery(ChannelTransport)
  -> ClusterAdmissionBridge
  -> RaftMetadataRuntime
  -> committed RaftMetadataCommand
```

The bridge records admission diagnostics and rejects stale discovered
generations before they reach the Raft proposal path.

Membership changes are now observable through a bounded event stream. Built-in
control planes expose `subscribe_membership()`, and client/member cache
runtimes expose `subscribe_cluster_membership()`. Slow subscribers receive lag
errors instead of blocking admission.

The sandbox exposes the same path through:

```text
POST /demo/cluster/real-adapters/run
```

The route uses chitchat `ChannelTransport`, `ClusterAdmissionBridge`, and
`RaftMetadataRuntime` to return discovery, bridge, Raft snapshot, committed
command, timeline, and pass/fail reports without requiring Docker or UDP
network setup.

## Cross-Process Invalidation Transport Spike

`CacheInvalidationFrame` is the first binary envelope for moving invalidation
intent across a process boundary. It carries:

- encoding version
- optional transport message id
- optional logical cluster name
- source node id
- source generation
- invalidation kind: key, tag, or flush

`InMemoryFramedInvalidationBus` is an in-process test/demo transport that
serializes every invalidation message into frame bytes before delivery. It is
not a durable or networked transport, but it exercises the same encode/decode
boundary future TCP, Redis, NATS, or Postgres adapters will use.

Transport diagnostics now include
`distributed_invalidation_decode_errors` in addition to lag, publish failures,
and closed receivers. Decode errors do not kill the background listener; the
bad frame is counted and the receiver continues.

## Design Direction

The implementation follows the documented plan:

```text
discovery finds candidates -> Raft commits control-plane metadata -> invalidation bus stays on the hot path
```

The in-memory runtime intentionally mirrors that model at a small scale:
generation protects against stale processes, epoch represents committed
metadata movement, and the invalidation bus remains separate from membership
coordination.
